## Project Overview

A shared travel itinerary planner for a 45-day South America trip (Peru, Bolivia, Chile) used by two people (Adam & Emma). Hosted on Azure Static Web Apps with Azure Blob Storage for file attachments and itinerary data.

| Build | **Vite** |
| Backend API | **Azure Functions** (Node.js, managed by SWA) |
| Storage | **Azure Blob Storage** (itinerary JSON + file attachments) |
| Hosting | **Azure Static Web Apps** (Free tier) |

| CI/CD | **GitHub Actions** (`.github/workflows/deploy.yml`) |

## Data Model

```typescript
interface Itinerary {
  title: string;              // "Peru • Bolivia • Chile"
  startDate: string;          // "2026-06-27" — fixed, all dates computed from this
  travelers: string[];        // ["Adam", "Emma"]
  days: Day[];                // always 45 items, order = trip order
}

interface Day {
  id: string;                 // stable UUID for drag tracking
  country: "peru" | "bolivia" | "chile" | "travel" | "home";
  location: string;           // e.g. "🇵🇪 Lima → Paracas"
  activities: Activity[];
  links: Link[];
}

interface Activity {
  id: string;
  text: string;               // e.g. "🚌 07:00 Peru Hop Stage 1"
  isHighlight: boolean;       // renders with amber highlight badge
}

interface Link {
  id: string;
  label: string;              // e.g. "Alpes Lima Kennedy Park"
  url: string;
  type: "accommodation" | "tour" | "transport" | "restaurant" | "other";
}
```

### Key Rules
- **Dates are never stored.** They are computed: `day[i].date = startDate + i days`
- **Day number** = array index + 1
- **Weekday** = computed from the date
- **Week grouping** = `Math.floor(index / 7)`
- **Country is a property of the day content**, not derived from position
- **Trip is always 45 days.** No adding/removing days.

## Date Computation

```typescript
const START_DATE = new Date('2026-06-27');

function getDayDate(index: number): Date {
  const d = new Date(START_DATE);
  d.setDate(d.getDate() + index);
  return d;
}
```

## UI Modes

### View Mode (default)
- Clean read-only cards
- Paperclip button for attachments
- "✏️ Edit" button in header

### Edit Mode
- Toggled via header button: "✏️ Edit" ↔ "✓ Done"
- Cards show: drag handles (⠿), editable fields, +/× buttons, country selector
- Amber glow border on page to indicate edit state
- Auto-save (debounced 500ms after last change)
- SaveIndicator in header: "Saving..." → "Saved ✓"

## Drag & Drop Behaviour
- **Whole days** are draggable (not individual activities)
- Drag only via handle icon (⠿), only in edit mode
- On drop: array reorders, all dates recalculate
- **Country boundary check**: if dropped day's country differs from neighbours → confirmation dialog:
  - "Moving '[location]' ([country]) between [neighbour country] days."
  - Options: **Keep [original country]** / **Change to [neighbour country]** / **Cancel**

## API Endpoints

| Method | Path | Purpose |
|--------|------|---------|
| GET | `/api/itinerary` | Load itinerary JSON from blob |
| PUT | `/api/itinerary` | Save itinerary JSON to blob |

| POST | `/api/get-upload-url` | Get SAS upload URL for a file |

## Conventions
- Use Svelte 5 runes syntax (`$state`, `$derived`, `$effect`)
- Use TypeScript strict mode
- Tailwind for all styling — no separate CSS files except `app.css` for Tailwind directives
- Component files: PascalCase (e.g. `DayCard.svelte`)
- Lib files: camelCase (e.g. `dates.ts`)
- API functions: kebab-case folders (e.g. `get-itinerary/`)
- IDs: use `crypto.randomUUID()` for new items
- No external state management — Svelte's built-in reactivity is sufficient
